    GEMINI_API_KEY — whichever matches LLM_PROVIDER in .env).

    Step 1 — Start the server from the backend/ directory:
      cd backend
      uvicorn app.main:app --port 8000

    Expected startup output (confirm each):
    - No import errors or AttributeError during startup
    - "Application startup complete" or equivalent uvicorn startup message
    - No "FAISS index is not loaded" error (index must load in lifespan)

    Step 2 — In a second terminal, run health check:
      curl -s http://127.0.0.1:8000/health
    Expected: {"status":"ok"}

    Step 3 — Check index status:
      curl -s http://127.0.0.1:8000/api/index/status | python -m json.tool
    Expected: JSON with "index_loaded": true and settings fields present

    Step 4 — Send a real chat question (use a question answerable from student_manual_2019.pdf):
      curl -s -X POST http://127.0.0.1:8000/api/chat \
        -H "Content-Type: application/json" \
        -d '{"question":"What are the requirements for shifting courses?","history":[]}' \
        | python -m json.tool
    Expected: JSON response with:
    - "answer" key containing a non-empty string (several sentences about shifting requirements)
    - "sources" key containing a list with at least one item (source citation with page number)

    Step 5 — Send a follow-up question (tests query rewriting with injected provider):
      curl -s -X POST http://127.0.0.1:8000/api/chat \
        -H "Content-Type: application/json" \
        -d '{"question":"What about the GPA requirement?","history":[{"role":"user","content":"What are the requirements for shifting courses?"},{"role":"assistant","content":"The requirements include..."}]}' \
        | python -m json.tool
    Expected: JSON response with "answer" and "sources" — the follow-up is contextually resolved

    Step 6 — Stop the server with Ctrl+C and confirm it exits cleanly (no hung event loop).
  </how-to-verify>
