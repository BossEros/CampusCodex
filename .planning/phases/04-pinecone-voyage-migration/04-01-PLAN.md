# Pinecone + Voyage Migration Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` or `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace the local FAISS and sentence-transformers retrieval stack with a production-ready managed retrieval stack using Pinecone for vectors and Voyage for embeddings and reranking, while preserving cited answers and keeping the project aligned with the existing roadmap.

**Architecture:** Keep the current layered RAG pipeline and existing provider pattern. Introduce a real retrieval seam first, then swap concrete adapters behind it. Preserve the two-stage Retrieval Pipeline (`Retrieval -> Reranking`) and keep `benchmark` isolated from `shared_kb` at the storage layer.

**Tech Stack:** FastAPI, Pinecone serverless, Voyage API, Pydantic settings, existing `LlmProvider` factory pattern, pytest/unittest, LangChain document types where still useful.

---

## Summary

This plan is intentionally **migration-first**. It does not try to implement the full admin ingestion product in one pass. The objective is to make the retrieval substrate production-ready first, then leave the codebase in the right shape for the later document-management phase.

The work is split into slow, trackable phases so the team can implement and verify one production concern at a time:

1. Establish a retrieval seam so the app is no longer typed directly against `FAISS`
2. Add embedding and reranker provider abstractions that mirror the existing LLM provider pattern
3. Introduce Pinecone as the managed vector adapter and switch runtime retrieval to it
4. Rebuild the benchmark corpus into the correct Pinecone namespace and verify source-cited answers still work
5. Remove heavy local ML dependencies and update docs/ops guidance

**Status note:** Complete and live-verified.

`backend/app/scripts/seed_pinecone_corpus.py` reuses the existing `pdf_loader.py`/`text_chunker.py` pipeline unchanged, embeds each chunk once via the configured `EmbeddingProvider` (Voyage), and upserts the vectors into one or more Pinecone namespaces. Metadata per vector: `title`, `source`, `page` (0-based), `page_number` (1-based), `chunk_text`, `chunk_index` â€” matching exactly what `PineconeVectorStoreAdapter` and `chat_service.build_sources()` expect for citations.

Voyage's free/no-payment-method tier enforces a very low rate limit (3 RPM / 10K TPM), which the naive first version of the script did not survive. The script was hardened before the real seeding run:
- small batches (20 chunks/request) with a fixed delay between requests
- backoff-and-retry specifically on `voyageai.error.RateLimitError` (up to 6 attempts)
- a local disk cache (`data/cache/voyage_chunk_embeddings.json`, gitignored) keyed by a SHA-256 fingerprint of the chunk text, so a re-run resumes from the last successfully embedded batch instead of re-embedding and re-spending rate-limit budget from scratch

Live seeding run: 150 PDF pages -> 412 chunks -> 412 embeddings -> upserted into both `benchmark` and `shared_kb`. `describe_index_stats()` confirmed:
```
namespaces: {'benchmark': {'vector_count': 412}, 'shared_kb': {'vector_count': 412}}
total_vector_count: 824
```
Eval (`benchmark`) and app (`shared_kb`) are separate namespaces in the same index, per the Phase 0 baseline â€” no vector mixing.

End-to-end verification against the live `/api/chat` endpoint (Pinecone-backed retrieval, real data) confirmed all four representative scenarios, each returning correctly page-cited answers from `Student Manual 2019`:
1. **Direct factual** â€” "What is the passing grade at CIT?" -> answered 3.0/75, cited page 35
2. **Vague / query-rewrite-dependent** â€” "What about failing?" after a passing-grade exchange -> correctly resolved via history-aware query rewriting, cited pages 24/27/34/35
3. **Follow-up with history** â€” "What documents do I need for that?" after an enrollment-process exchange -> correctly asked for disambiguation given multiple matching document-requirement sections, cited pages 25/26/72
4. **No-answer / weak-context** â€” "What is the capital of France?" -> correctly declined, stating the student manual doesn't cover it, despite Pinecone still returning low-score fallback matches (score ~0.33-0.35)

One unrelated finding surfaced during verification: the `GROQ_API_KEY` in `backend/.env` is currently invalid (`groq.AuthenticationError: Invalid API Key`), which is a pre-existing credentials issue unrelated to this migration â€” not fixed here. Verification instead used `LLM_PROVIDER=anthropic` (env override, `.env` left untouched) to exercise the full answer-generation path; retrieval/reranking/citation behavior is provider-agnostic and identical regardless of which `LlmProvider` is used.

Full suite: `python -m pytest` -> 26 passed, 0 failed (unchanged from Phase 3; no test changes were needed for Phase 4's script/docs work).

## Explicit Assumptions and Defaults

- This plan is **migration-first**, not the full document-management implementation
- The plan lives inside `.planning` because this repo already uses that as the team tracking system
